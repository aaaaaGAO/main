## 1. 前端 GUI 模块

**功能说明**  
提供用户可见的 Web 界面与交互：页面加载、配置表单展示、路径选择（含 Tkinter 文件/目录选择弹窗）、按域（左右后/中央/DTC）触发生成、执行状态与结果展示、错误详情展示。前端不包含具体生成逻辑，仅负责请求转发与结果呈现。

**对应代码位置**  
- **入口**：`app.py`（Flask 应用、心跳、Tk 弹窗挂载
- **Web 层**：`web/__init__.py`（create_app、蓝图注册）、`web/routes/common.py`（心跳、文件选择、配置加载、筛选项、一键生成、预设）、`web/routes/lr_rear.py`（左右后域 CAN/配置）、`web/routes/central.py`、`web/routes/dtc.py`  
- **前端资源**：`templates/index.html`（主页面）、`static/js/api.js`（后端 API 封装）、`static/js/config_handler.js`（配置状态与预设）、`static/js/ui_controls.js`（Tab、弹窗、运行按钮、心跳）  
- **GUI 服务**：`services/gui_service.py`（Tkinter 弹窗、文件/文件夹结构解析）

**验收要点**  
- 浏览器可正常打开界面，各域 Tab 与按钮可点击  
- 文件/目录选择弹窗能正确返回路径并回填到配置  
- 点击「开始运行」后能触发对应域生成任务，并返回成功/失败与详情  
- 配置保存、加载、筛选项（等级/平台/车型等）与前端联动正常

## 2. io-mapping 解析及中间文件生成模块

**功能说明**  
从 `Configuration.txt` 的 `[IOMAPPING]` 读取 IO_mapping Excel 路径与 Sheet 配置，解析 Excel 中的 Name/Path/Values 列，构建「信号名→路径」「枚举值→数值」等映射，供步骤解析时做参数替换与枚举翻译。会生成 `IO_Mapping.log` 等解析过程日志；不直接生成最终 .can/.cin，而是为 CAN/CIN 生成提供“中间”上下文（IOMappingContext）。

**对应代码位置**  
- `core/translator/io_mapping.py`：`load_io_mapping_from_config`、`IOMappingContext`、`transform_args`  
- 配置：`Configuration.txt` → `[IOMAPPING].Inputs`（如 `input/xxx.xlsx | *`）

**验收要点**  
- 配置正确时能成功加载 IO_mapping Excel，无报错  
- 步骤中的信号名（如 `J_xxx+`）能被正确替换为 Path；Values 枚举翻译符合配置  
- 表头错误、Name 冲突等能写日志并在生成阶段得到合理处理（报错或跳过）  

---

## 3. did-config 解析及中间文件生成模块

**功能说明**  
解析 DID 配置相关输入（Configuration 中指定的 DID 配置 Excel），按 Sheet/Name/Byte/Bit 等结构解析，为生成 `DIDConfig.txt`（或配置指定文件名）提供中间数据；与 Config Enum 枚举翻译配合，供 DID 配置生成器使用。

**对应代码位置**  
- `generators/capl_didconfig/`：`runtime_io.py`（DID 配置表解析）、`runtime.py`（主流程）、`service.py`（对外入口）  
- `generate_did_config.py`：入口脚本，调用 `DIDConfigGeneratorService.run_legacy_pipeline()`  
- 配置：`Configuration.txt` → `[DID_CONFIG].input_excel` 等  

**验收要点**  
- 指定 DID 配置 Excel 与输出路径后，能正确解析表头与数据行  
- 生成的 DIDConfig 输出文件格式、内容符合预期  
- Web 界面 DID 配置任务可正常触发并完成  

---

## 4. reset-did（DID 信息）解析及中间文件生成模块

**功能说明**  
**Reset-DID 相关数据在本项目中由「DID 信息（DIDInfo）」模块统一实现。** 配置项 `[LR_REAR].didinfo_inputs` 指向的 Excel 即「ResetDid_Value 配置表」；DIDInfo 生成器读取该表及 DID 信息相关 Excel，按 Configure DID、Length、Byte、Bit 及车型列解析，生成 `DIDInfo.txt`（或配置指定路径）。因此无单独的「Reset-DID 独立管线」，Reset-DID 与 DID 信息输出一并在本模块验收。

**对应代码位置**  
- `generators/capl_didinfo/`：`runtime.py`（主流程、配置加载）、`runtime_io.py`（表头 Configure DID/Length/Byte/Bit、车型列、按 Sheet/车型生成）、`service.py`（对外入口）  
- `generate_didinfo_from_excel.py`：入口脚本  
- 配置：`Configuration.txt` → `[LR_REAR].didinfo_inputs`（ResetDid_Value 配置表路径）、`output_dir_didinfo`、`didinfo_variants` 等  

**验收要点**  
- 配置好 didinfo_inputs（ResetDid_Value 配置表）及输出路径后，能正确解析表头与数据  
- 生成的 DIDInfo 输出文件格式、内容符合预期  
- Web 界面 DID 信息（含 Reset-DID 数据源）生成任务可正常触发并完成

## 5. SOA 通信矩阵解析及中间文件生成模块

**对应代码位置**  
- 当前：无独立 `generators/capl_soa/` 或等价模块  

**验收要点**  
- 验收时可在总表中标注为「不适用」或「规划中」

## 6. 串口通讯矩阵解析及中间文件生成模块

**功能说明**  
解析串口（UART）通信矩阵配置与 Excel（如 IVIToMCU、MCUToIVI 等 Sheet），生成 UART 配置中间数据并输出为 `Uart.txt`（或配置指定路径），供车载/台架串口通信使用。

**对应代码位置**  
- `generators/capl_uart/`：`runtime_io.py`（UART 矩阵表解析、Uart.txt 生成）、`runtime.py`（主流程）、`service.py`（对外入口）  
- `generate_uart_from_config.py`：入口脚本  
- `core/parse_table_loggers.py`：`get_uart_matrix_logger`（UART 矩阵解析日志）  
- 配置：`Configuration.txt` 中 UART 相关 Section（路径、FrameType、串口参数等）  

**验收要点**  
- 配置正确的 UART 矩阵 Excel 与输出路径后，能解析表头与数据  
- 生成的 `Uart.txt` 格式、帧类型与字段符合预期  
- Web 界面 UART 生成任务可正常触发并完成

## 7. 关键字用例解析及 CAN 文件生成模块

**功能说明**  
读取测试用例 Excel，按「关键字」解析步骤行（结合 StepParser、IO Mapping、Config Enum），将每条步骤翻译为 CAPL 函数调用，再按 CAN 用例模型拼装为 `.can` 文件（含 testcase/teststep、includes 等）。包含用例 ID 清洗、去重、等级/平台/车型过滤及生成日志。

**对应代码位置**  
- `generators/capl_can/`：`excel_repo.py`（读 Excel→CANTestCase）、`translator.py`（步骤翻译）、`renderer.py`（.can 文本拼装）、`runtime.py`、`runtime_io.py`、`service.py`、`models.py`  
- `core/parser/step_parser.py`：`parse_step_line`（关键字匹配、参数翻译）  
- `core/translator/`：io_mapping、config_enum、keyword_mapping  
- `generate_can_from_excel.py`：入口脚本  

**验收要点**  
- 给定合规 Excel 与配置，能稳定生成 .can 及 Master.can（若启用）  
- 步骤中的关键字、信号名、枚举值能正确翻译为 CAPL 代码  
- 用例 ID 清洗、重复处理、过滤逻辑符合业务规则；解析/生成日志可追溯

## 8. 基于 CAN 生成 XML 文件模块

**功能说明**  
以与 CAN 同源或同结构的测试用例 Excel 为输入，按用例 ID、名称、等级、功能模块等分组，生成 XML 格式的测试模块文件（.xml），供测试管理或执行平台使用。本模块**独立读 Excel 并写 XML**，不依赖已生成的 .can 文件。

**对应代码位置**  
- `generators/capl_xml/`：`runtime_io.py`（读 Excel、用例 ID 去重与清洗、写 XML）、`runtime.py`（主流程）、`service.py`（对外入口）  
- `generate_xml_from_can.py`：入口脚本  
- `core/common/sanitizer.py`：用例 ID 清洗（被 capl_xml 使用）  

**验收要点**  
- 指定 Excel 与 XML 输出路径后，能正确解析表头与行数据  
- 生成的 XML 结构、层级、属性符合约定（如按等级/模块分组）  
- Web 界面 XML 生成任务可正常触发并完成

## 9. Clib 关键字集合解析及 CIN 文件生成模块

**功能说明**  
解析 Clib 关键字集合（来自 Clib 矩阵 Excel 或配置），结合步骤解析引擎与 IO Mapping/Config Enum，生成 `.cin` 格式的 CAPL 库文件，供 CAN 测试脚本 include 使用。

**对应代码位置**  
- `generators/capl_cin/`：`runtime.py`、`runtime_io.py`（Clib 矩阵/步骤解析、CIN 输出）、`service.py`、`constants.py`  
- `generate_cin_from_excel.py`：入口脚本  
- `core/parser/step_parser.py`、`core/translator/`：与 CAN 共用步骤解析与映射  

**验收要点**  
- 给定 Clib 矩阵/用例 Excel 与配置，能稳定生成 .cin 文件  
- 关键字、参数翻译与 CIN 导出函数名（如 `g_HIL_Clib_Swc_Clib_*`）符合约定  
- Web 界面 CIN 生成任务可正常触发并完成

## 10. 工程配置保存及导入模块

**功能说明**  
管理工程级配置：读取/写入 `Configuration.txt`、`FixedConfig.txt`，去重与格式化；提供前端所需的配置项（路径、筛选项、域配置等）；支持保存当前界面配置、导入已有预设；与 ConfigManager 的线程安全（RLock）及 sync_to_file 等行为一致。

**对应代码位置**  
- `services/config_manager.py`：ConfigManager，INI 读/写/去重/格式化、load_ui_data、save_ui_data、update_domain_config、sync_to_file  
- `services/config_service.py`：对前端暴露的配置接口（路径、筛选项、预设等）  
- `services/filter_service.py`：解析 `filter_options.txt`，提供等级/平台/车型等下拉数据  
- `utils/config.py`：底层 INI 读取（含重复 option 容忍）  

**验收要点**  
- 界面修改配置后保存，能正确写回 `Configuration.txt`（及约定的 FixedConfig 覆盖逻辑）  
- 加载/导入预设后，界面展示与文件内容一致  
- 多域（LR_REAR、中央、DTC）的配置项互不串扰

## 11. log 日志模块

**功能说明**  
统一管理运行期日志：按任务创建 `log/log_YYYYMMDD_HHMMSS/` 及子目录（生成文件日志、解析表格日志）；Tee 将 stdout 重定向到日志；各解析表（IO_Mapping、TestCases、Uart_Matrix、caseid_clean_dup 等）有专用 Logger；支持同进程去重（caseid）、handler 清理与下次运行新目录。

**对应代码位置**  
- `utils/logger.py`：TeeToLogger、日志级别、log_progress_or_info 等  
- `core/generator_logging.py`：生成器专用日志初始化、目录创建、handler 管理  
- `core/log_run_context.py`：ensure_run_log_dirs、reset_run_context（任务级重置）  
- `core/parse_table_loggers.py`：get_testcases_parse_logger、get_uart_matrix_logger、get_caseid_clean_dup_logger 等  
- `core/caseid_log_dedup.py`：DedupOnceFilter、reset_dedup_filter  

**验收要点**  
- 单次生成任务产生的日志落在同一时间戳目录下，且内容完整  
- 二次点击「运行」会生成新的时间戳目录，不沿用旧 handler  
- 控制台/文件中的 ERROR、关键步骤可追溯；解析表日志（如 caseid_clean_dup.log）无异常重复或丢失

## 12. 打包模块

**功能说明**  
使用 PyInstaller 将项目打包为单文件 EXE（如 `测试用例生成工具_2026.x.x.exe`），包含 Flask、openpyxl、各 generate_* 入口及 core/utils/generators 等依赖；打包时纳入 templates/、static/ 及约定资源，使 EXE 在无 Python 环境下可独立运行并打开 Web 界面。

**对应代码位置**  
- `build_exe.py`：PyInstaller 参数、hidden_imports（含 core.*、各生成器）、add-data、exe 名称与输出路径  

**验收要点**  
- 在目标环境执行 `python build_exe.py` 能成功生成 EXE，无缺失模块报错  
- 双击 EXE 能启动并打开浏览器访问界面，配置与生成功能与开发环境一致  
- 打包后无 Python 的工位可正常运行，路径与配置文件可写可读
