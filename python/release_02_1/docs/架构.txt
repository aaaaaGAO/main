二、各目录/模块设计说明
--------------------------------------------------------------------------------

【根目录脚本】
  - app.py：程序入口。打包后 exe 或桌面启动用，负责创建 Flask 应用、心跳监控、端口探测、自动打开浏览器。
  - build_exe.py：一键打包为单文件 exe，纳入 templates/static/ 及必要配置，便于交付无 Python 环境用户。
  - 生成任务入口已收口至 generators 各子包内的 entrypoint 模块（无根目录 generate_*.py），Web/任务编排通过 TaskService 调用 generators.*.entrypoint.run_generation / run_cli 等接口，避免根目录脚本散落与误触。

【web/】
  - 职责：HTTP 路由与请求分发，不包含业务算法。create_app 创建 Flask 应用并注册 common、lr_rear、central、dtc 四个蓝图。
  - routes/common.py：通用 API（心跳、健康检查、文件选择、配置加载、筛选项、一键生成、预设保存/导入），供前端 api.js 调用。
  - routes/lr_rear.py、central.py、dtc.py：按域提供“生成”与“保存配置”接口，内部调用 services 层，避免在路由里写复杂逻辑。

【services/】
  - 职责：应用服务层，组合领域逻辑，供 Web 调用。把“配置读写”“任务编排”“弹窗与文件解析”“运行前校验”从 Web 中剥离，便于测试与复用。
  - config_manager：统一读写主配置文件、固定配置文件，保证格式一致、去重、同步。
  - config_service：对前端暴露的配置接口（路径、筛选项、预设、域配置），与 config_manager 配合。
  - task_orchestrator：按域（左右后/中央/DTC）编排“先校验再依次执行 DIDConfig、DIDInfo、CIN、UART、CAN、XML”等任务；同一任务内复用 workbook 对象，降低重复解析开销，并在 finally 阶段统一关闭。
  - task_service：单任务执行（CAN/XML/CIN/DIDInfo/DIDConfig/UART），调用对应 generators.*.entrypoint.run_generation / run_cli。
  - gui_service：Tkinter 弹窗（文件/文件夹选择、另存为）、文件/文件夹结构解析（Excel sheet、CAN testcase、XML testgroup）。
  - filter_service：解析 config/filter_options.ini，提供等级/平台/车型等下拉数据（缺文件时返回空列表；仅支持此文件名，无 .txt 回退）。
  - run_validator：运行前校验输出目录可写、输入路径存在，避免执行到一半才报错。

【core/】
  - 职责：领域逻辑层，纯业务算法，不依赖 Web/GUI。被 generators 与 services 复用，保证“同一套解析/过滤/翻译规则”处处一致。
  - base_task、case_filter、generator_config、mapping_context：为各生成器提供“配置读取、过滤、映射上下文”等基础能力。
  - parser/step_parser、translator/（io_mapping、config_enum、keyword_mapping）：步骤行解析与参数/枚举翻译，CAN 与 CIN 共用。
  - generator_logging、run_context、log_run_context、parse_table_loggers、caseid_log_dedup：日志与运行上下文管理，避免多任务串状态。
  - step_error_detail、keyword_error、error_module：统一错误明细与模块归类，便于前端与日志展示。
  - common/：通用小工具（用例 ID 清洗、Clib 名称清洗、生成汇总）+ 少量统一导出入口。

【generators/】
  - 职责：具体格式转换实现。每个子包（capl_can、capl_cin、capl_xml、capl_didinfo、capl_didconfig、capl_uart）只关心“自己这种输出格式”的编排与写出，读表、过滤、步骤解析等复用 core/infra。
  - 入口约定：各子包提供 entrypoint 模块，对外暴露 run_generation / run_cli（及别名），供 TaskService 与命令行（python -m generators.capl_can.entrypoint）调用；具体编排与实现仍在 runtime、runtime_io、service 中。
  - 域隔离约定：各 runtime 仅读取当前 domain 对应 section，缺失即报错；已移除 get_first 跨节兜底与 runtime 内对 [PATHS] 的跨域读取。
  - 结构收口约定：CAN/XML/UART 的 runtime_io 已完成“散函数命名收口 + Utility 统一导出”，外部调用保持稳定，内部实现更集中。
  - 设计目的：新增一种输出格式时，只需新增一个 generator 子包并在其内实现 entrypoint.run_generation（或 run_cli），不改动 core 与 Web。

【infra/】
  - 职责：基础设施层，技术细节。与“业务规则”解耦，便于替换实现（如换 Excel 库、换路径约定）。
  - excel/workbook.py：openpyxl 工作簿打开、合并单元格取值等硬核逻辑；engine 为统一入口；header 为表头/列定位。
  - filesystem/pathing.py：get_base_dir、get_project_root、find_config_path，统一路径解析（含打包 frozen 场景）。
  - config/：配置项读取、Inputs 行解析（path | sheet）。

【utils/】
  - 职责：通用工具薄封装，各层复用。字符串处理、INI 读取、日志级别、路径解析、sheet 勾选解析等；Excel 读写中与 openpyxl 强相关的部分已迁至 infra.excel，utils 仅保留通用能力与统一导出。

【static/js/】
  - api.js：封装所有与后端的 fetch 请求（文件选择、解析结构、加载配置、筛选项、生成、预设、心跳等），供页面与 config_handler、ui_controls 调用。
  - config_handler.js：配置恢复、自动保存、预设导入导出；维护当前选择与表单状态（selection、各 config 对象），提供 collectCurrentState、getChecks、getSelectedSheets 等。
  - ui_controls.js：Tab 切换、路径选择弹窗、用例树勾选、运行按钮、心跳定时；依赖 api.js 与 config_handler.js，实现用户可见的交互逻辑。

【templates/】
  - index.html：单页主界面，包含左右后/中央/DTC 三个 Tab、配置表单、文件与目录选择、开始运行、结果与错误展示，并引用 static/js 三个脚本。

【docs/】
  - 存放架构说明、函数注释规范、验收清单说明、必需文件说明等，便于交付与验收。

【config/ 下配置文件】
  - 主配置文件、固定配置文件为运行所需；`filter_options.ini` 为**可选**（无则前端「等级/平台/车型」等下拉链表为空），且**仅**识别 `filter_options.ini`（与代码 `resolve_filter_options_path` 一致，不支持 `filter_options.txt`）。主/固定配置可选用 `Configuration.ini`/`FixedConfig.ini`（对历史 `*.txt` 的说明见《必需文件说明.txt》）。文件通常放在工程根下 `config/`，与 exe 同目录时即 exe 同级 `config/`。
