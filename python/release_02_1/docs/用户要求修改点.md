3. 所有的生成中间文件相关的，换成代码模板的方式，jinja2客户已取消）
4. 变量命名，不要使用python库函数，比如变量_write_formatted_config函数中变量path等（不止这一个）；变量命名根据实际含义进行命名，不要使用一个字母'o'/'p'/'v'等（不止这一个）；自己整体看下代码，涉及到这类问题的全部修改-
5. 函数命名混乱，'_'开头的函数还在外部模块进行调用。等等此类函数命名问题全部解决
6. config中的txt，换成ini文件进行配置解析保存，除注释外，不要用中文
7. 大量的常量，放在模块顶部进行定义和配置，如_apply_state_to_config里面的大量常量等
8. 相同的逻辑多次使用，抽离函数，如_apply_state_to_config等函数里面很多雷同的逻辑。且函数该拆分需要拆分
9. 尽量（非必要）不要在函数内部再定义内部函数，如_is_configured_c_pwr等
10. 路由请求等由单独的模块管理，内部调用的复杂处理逻辑封装到别的模块进行调用
11. 保存配置基于读取出来的，修改保存就行，不要再逐行写；配置文件结构要固定，不需要的值置空，解析的时候做判断逻辑
12. 大量相同的常量，通过配置模块进行管理，如lr_keys等，出现过无数次，现在都是在每个函数中写死的。这些东西，如果要改逻辑，达到效果一定是只改一处
13. 函数不要散放在模块内，该封装成工具类的就封装
