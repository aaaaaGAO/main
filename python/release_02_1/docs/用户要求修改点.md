原始需求全文（备忘第二版）：
1.左右域，不需要生成uart相关内容，逻辑去掉。。生成哪个界面的内容，该界面一定会配置，不需要去别的界面区域块拿数据，一定是独立的，包括后台的解析或生成；设计配置的时候，分块分级（大块下面再是小块，比如左右域一级，然后下面二级就是左右域需要配置的东西，具体看ini文件格式设计），生成或者解析哪块，就使用或填充哪块数据。不仅仅是配置区分，各块前/后端的解析/生成等逻辑也区分开来。现在混在一起太乱
2.各文件生成模块，全为散的函数（散的函数问题要解决，这种直接的函数调用的都可以通过封装成工具类去调用），函数命名为main等问题
3.输入文件和代码和生成的文件类型需要匹配上，可能是之前的命名，如代码中的didinfo对应的是现在前端界面输入的resetdid相关内容
4.除开点击生成按钮后存储的Log，工具在运行本身（点击生成之前），代码也需要记录日志，不仅仅是print
5.gconfig.get_first等配置处理，直接到对应的配置节点拿信息，没获取到就报错，而不是按顺序获取非空配置。各不同域的配置解析生成等，逻辑不要混在一起，这个多次提出来一定注意拆分
6.生成generated_from_cases.can逻辑拆分
7.代码中工作目录切换逻辑，改为使用路径拼接方式，同文件路径配置，通过单独一个py进行配置，path相关的函数，封装成一个类，再进行函数封装和调用。比如获取工程路径等
8.同一个操作对应的接口中不同生成功能，反复在读同一份配置文件或者excel等，重复操作重复处理
9.中央域必要输入文件未做校验
10.服务通信矩阵解析未合入当前web代码中；左右域/DTC界面，均需要导入soa通信矩阵
11.根据需求新增arxml自动string类型转换工具，独立封装打包
12.函数调用缺少入参
13._restore_config_after_dtc_did_cin等函数，各个界面的配置/前后端解析生成代码等不独立
14.read_uart_excel_data等重复load。同同一个操作对应的接口中不同生成功能，反复在读同一份配置文件或者excel等，重复操作重复处理一样
