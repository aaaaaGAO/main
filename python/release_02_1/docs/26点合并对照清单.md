# 客户要求 26 点合并对照清单（当前代码库）

1.import导入语句，现在有很多放在函数内部，按官方风格统一放在文件头部。
2. 文件路径配置，通过单独一个py进行配置，path相关的函数，封装成一个类，再进行函数封装和调用。比如获取工程路径等。
3. 变量命名，不要使用python库函数，比如变量_write_formatted_config函数中变量path/keys/get等等（不止这一个）；变量命名根据实际含义进行命名，尽量不要使用一个字母'o'/'p'/'v'等（不止这一个）；自己整体看下代码，涉及到这类问题的全部修改-。
4. 函数命名混乱，'_'开头的函数还在外部模块进行调用。等等此类函数命名问题全部解决。
5. config只是config文件夹下的的txt，换成ini文件进行配置解析保存，生成物就是txt不影响。
6.大量的常量，放在模块顶部进行定义和配置，如_apply_state_to_config里面的大量常量等。
7. 相同的逻辑多次使用，抽离函数，如_apply_state_to_config等函数里面很多雷同的逻辑。且函数该拆分需要拆分。
8.尽量（非必要）不要在函数内部再定义内部函数，如_is_configured_c_pwr等。
9. 路由请求等由单独的模块管理，内部调用的复杂处理逻辑封装到别的模块进行调用。
10. 保存配置基于读取出来的，修改保存就行，不要再逐行写；配置文件结构要固定（现在代码里面一直反复的add/remove配置节点），不需要的值置空，解析/生成的时候做判断逻辑。

3. **`@classmethod` 使用约定**（与客户准则及**截图 3** 对齐；**文档 + 主链代码一致**）  
   下列为**合法**使用 `classmethod` 的三大场景；除此之外优先 **实例方法** 或 **`@staticmethod`**：  
   - **替代构造函数**（Alternative Constructors）：如 ``from_config()``、``from_base_dir()``、`ProjectPaths.from_base_dir`。  
   - **访问或维护类级属性**：读/写**类变量**（非实例字段）时。  
   - **支撑继承**：子类可能重写该方法，且父类需绑定**当前类**（``cls``）时。
